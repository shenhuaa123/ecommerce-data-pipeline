# E-commerce Data Pipeline

这是一个基于PySpark的电商数据处理与分析项目。

项目从模拟业务数据生成开始，完成数据清洗、订单宽表构建、事实表与维度表建模、数据质量检查和多维业务分析，并将处理结果导出为CSV文件。

当前模拟数据规模：

- 300,000 条订单
- 30,000 名客户
- 150 个商品
- 30 个地区

项目使用PySpark本地模式运行，主要用于实践完整的批处理数据管道和Spark DataFrame数据处理流程。

## 数据处理流程

```text
生成模拟数据
        ↓
读取原始CSV
        ↓
数据清洗与转换
        ↓
构建订单宽表
        ↓
构建事实表与维度表
        ↓
数据质量检查
        ↓
多维业务分析
        ↓
导出CSV结果
```

完整Pipeline入口：

`src/pipeline.py`

数据生成入口：

`src/generate_data.py`

## 主要实现

### 1、数据生成与清洗

项目自动生成订单、客户、商品和地区数据。

模拟数据根据商品类别、价格、复购等级和商品类型等规则设置不同的生成权重，并主动加入少量数量异常、负价格和订单状态缺失等数据，用于验证后续清洗和质量检查流程。

数据清洗主要包括：

- 数据类型转换
- 缺失值处理
- 异常值过滤
- 非法枚举过滤
- 主键去重
- 成交价、销售额、成本和利润等业务指标计算

### 2、数据建模

清洗后的订单会与客户、商品和地区数据关联，形成订单宽表。

项目同时构建：

```text
fact_orders
dim_customers
dim_products
dim_regions
dim_date
```

形成以订单事实表为中心的星型数据模型。

### 3、数据质量检查

数据质量检查覆盖：

- 数据规模变化
- 主键唯一性
- 关键字段缺失
- 异常值和业务规则
- 多表关联完整性
- 事实表与维度表一致性

检查结果按照 `LOW`、`MEDIUM` 和 `HIGH` 进行风险分级，并输出风险摘要、完整报告和风险明细CSV。

### 4、业务分析

当前实现的分析包括：

- 商品与类别销售分析
- 地区销售分析
- 月度和年度销售趋势
- 折扣分析
- Top Customers
- 支付方式和订单状态分析
- 复购等级和商品类型分析
- 非现金商品成本分析
- 动态条件Top Customers分析

动态客户分析支持按地区、时间、商品类别和支付方式筛选，并可以按照销售额、利润、订单数、客单价或利润率排序。

## 补充说明

- 模拟数据生成阶段会主动注入少量异常数据，例如数量为 `0`、负价格和订单状态缺失，用于验证后续数据清洗和数据质量检查流程是否能够正确发现问题。
- 部分比例指标使用Window计算，是因为需要在聚合结果中保留各分组明细的同时计算全局总量和占比。当前Window主要作用于已经完成 `groupBy` 聚合后的少量结果数据，因此对当前项目规模影响较小。
- 普通销售分析只统计已完成订单中的普通商品；赠品、试用品和积分兑换商品单独统计成本，避免非现金商品影响正常销售额和利润指标。
- 项目同时保留订单宽表和星型模型：宽表便于直接进行分析和质量检查，事实表与维度表用于练习更清晰的数据仓库建模和多维分析结构。
- Spark原始输出通常包含分区文件，项目会进一步生成带UTF-8 BOM的单一CSV，主要是为了方便当前本地项目直接使用Excel查看结果。

## 环境

主要依赖：

- Python
- PySpark 4.1.1

安装依赖：

```bash
pip install -r requirements.txt
```

## 运行

### 1. 生成模拟数据

```bash
python src/generate_data.py
```

生成的原始数据保存在：

`data/raw/`

### 2. 执行完整数据管道

```bash
python src/pipeline.py
```

Pipeline会依次完成：

```text
原始文件检查
→ 创建 SparkSession
→ 加载原始数据
→ 数据清洗
→ 构建订单宽表
→ 构建事实表与维度表
→ 保存处理结果
→ 生成数据质量报告
→ 构建销售分析基础表
→ 执行业务分析
→ 保存分析结果
```

运行完成后，结果分别保存在：

- `data/processed/`：清洗结果和订单宽表
- `data/warehouse/`：事实表和维度表
- `data/output/`：数据质量报告和业务分析结果

## 项目范围

当前项目运行于PySpark `local[*]` 模式，主要关注批处理数据流程、DataFrame转换、数据质量检查、维度建模和业务分析。

项目暂未涉及 Spark 集群部署、实时流处理和自动化测试。

## 当前存在的问题 

项目目前可以完整运行，但仍存在以下限制和改进空间：

 - 项目使用PySpark `local[*]` 模式运行，尚未部署到真实Spark集群 
 - 项目只实现批处理流程，没有使用Spark Structured Streaming进行实时数据处理 
 - 输入CSV使用 `inferSchema` 自动推断字段类型，后续可以改为显式 `StructType` Schema，提高输入结构的稳定性 
 - 模拟数据生成未固定随机种子，因此不同运行批次的数据分布和清洗结果会存在一定差异 
 - 分析结果和仓库数据主要保存为CSV，后续可以增加Parquet等列式存储格式 
 - 为方便生成单个Excel可读CSV，部分输出使用单分区方式写出，不适合直接扩展到更大的数据规模 
 - 部分全局比例计算使用无分区Window，会产生Spark `WindowExec` 性能警告；当前参与计算的是聚合后的少量结果数据，对本项目规模影响较小 
 - 数据质量检查包含较多Spark Action，在更大数据规模下仍有进一步减少重复计算的空间 
 - 没有建立Unit Test和Pipeline Integration Test，现有数据质量检查主要用于验证数据内容和业务规则，不能替代自动化测试 
 - 数据完全由模拟生成器生成，尚未接入真实业务数据源、数据库或消息队列 这些问题不会影响当前项目完整运行，但如果继续扩展，可以从显式Schema、自动化测试、Parquet存储、真实集群运行和减少重复Spark Action等方向继续改进。
